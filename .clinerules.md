# DCE Architecture Review: Integration Framework

## Objective

You are acting as a Senior Software Engineer and Systems Architect for the Dynamic Criminal Ecosystem (DCE).

Your responsibility is to design and maintain a robust, extensible integration architecture that preserves the independence of the DCE simulation while allowing external resources to interact with it.

The goal is not simply to "support ERS" or "support Sonoran."

The goal is to create an integration framework that can support any current or future resource without requiring changes to the simulation itself.

---

# DCE Integration Philosophy

DCE is a persistent criminal world simulation.

It owns:

- Organizations
- Economy
- Territories
- AI
- Heat
- Intelligence
- Evidence
- Investigations
- Scenarios
- World State

DCE does **not** own:

- CAD systems
- MDT systems
- Dispatch resources
- Inventory systems
- Banking systems
- Phone systems
- Voice systems
- Target systems

These are external applications that consume information produced by DCE.

---

# Architectural Principles

The following principles are mandatory.

## 1. Simulation First

External integrations must never influence how the simulation operates internally.

The criminal world must continue functioning exactly the same whether:

- ERS is installed
- Sonoran CAD is installed
- PS-MDT is installed
- No integrations are installed

Adapters expose information.

They do not create simulation.

---

## 2. DCE Owns the Truth

Every piece of authoritative data belongs to DCE.

Examples:

Organization finances

Territory ownership

Heat

Evidence

Investigations

Scenario progression

Organization hierarchy

Supply chains

External systems display or synchronize this information.

They never become the source of truth.

---

## 3. Adapters are Translators

Adapters translate DCE domain events into external API calls.

They must never:

- contain AI
- generate gameplay
- modify organizations
- create scenarios
- alter economy
- bypass services

Their responsibility ends after successful translation.

---

## 4. Event Driven

Every integration begins with an Event Bus event.

Example

Scenario Escalated

↓

Dispatch Requested

↓

Dispatch Adapter

↓

ERS

or

↓

Sonoran

or

↓

Native Adapter

No service may directly communicate with an external framework.

---

# Integration Manager Responsibilities

Design an Integration Manager responsible for:

- discovering adapters
- loading adapters
- validating compatibility
- managing priorities
- providing fallbacks
- exposing adapter capabilities
- handling failures gracefully

The Integration Manager must not contain simulation logic.

---

Define common contracts for every adapter type.

Radio

Doorlock

Voice

Every implementation should conform to the same interface so that integrations can be replaced through configuration alone.

---

# Dispatch Philosophy

DCE does not generate scripted callouts.

Instead, dispatches emerge naturally from world events.

Example

Drug transaction

↓

Argument

↓

Fight

↓

Weapon drawn

↓

Shots fired

↓

Witness observes event

↓

911 call generated

↓

Dispatch Adapter

↓

ERS

Dispatch systems receive consequences of the simulation rather than predefined missions.

---

# Information Fidelity

Police and emergency services should only receive information that would realistically be available.

Information quality should depend on factors such as:

- witness visibility
- distance
- weather
- lighting
- panic level
- line of sight
- time delay
- neighborhood confidence
- witness reliability

The simulation knows everything.

Responders receive only what civilians perceive.

---

# Evidence Integration

Evidence is owned entirely by DCE.

Adapters synchronize evidence into external systems without changing its meaning.

Possible outputs include:

ERS inventory items

PS-MDT case attachments

Sonoran evidence

Native world props

Different presentation.

Same underlying evidence.

---

# Investigation Integration

Investigations remain inside DCE.

CAD and MDT systems visualize investigations.

External systems never determine investigative outcomes.

They provide tools for players, not authority over the simulation.

---

# Extensibility

The architecture must support future integrations without modifying existing services.

Adding support for a new CAD or MDT should require only:

- a new adapter
- configuration
- documentation

No changes to the simulation should be necessary.
