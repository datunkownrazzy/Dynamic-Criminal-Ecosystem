# DCE Glossary

**Status:** Accepted
**Version:** 1.0
**Owner:** Datunkownrazzy

---

Consistent vocabulary across code, docs, and conversation. When in doubt, use the term defined here rather than a synonym.

---

**Agent**
A simulated NPC controlled by DCE — a gang member, dealer, lookout, or civilian with observable behavior. (Not to be confused with "AI agent" in the general software sense, though the usage is related.)

**Organization**
A criminal group (street gang, cartel, motorcycle club, mafia family, etc.). Has resources, territory, personality weights, a leadership hierarchy, and a state (see *Organization State*).

**Organization State**
The current phase an organization is in: Dormant → Growing → Stable → Aggressive Expansion → Conflict → Under Investigation → Suppressed → Recovering → (loop). Determines which behaviors are available to it.

**Territory**
A defined zone/district that can be owned or contested by an Organization. Has its own lifecycle (Neutral → Claimed → Established → Prosperous → Contested → Violent → Police Crackdown → Recovered) and tracked values (influence, heat, civilian support, etc.).

**World State**
The current values describing a district or the whole map at a point in time — police presence, civilian density, gang influence, economic health, weather, time of day, etc. Primary input to the AI Director's scoring.

**Simulation Event**
Any discrete thing the simulation generates — a patrol, a drug sale, a territory shift, a robbery. Not to be confused with a codebase "event" in the pub/sub sense (see *Event Bus*); when ambiguity is possible, prefer "Simulation Event" explicitly.

**Scenario**
An instance of a criminal activity as it plays out — the specific drug deal, the specific robbery — including its current stage and outcome path. Built from an Event Escalation definition.

**Incident**
A Scenario that has escalated to the point of generating a Dispatch call and becoming player-interactive (Layer 2+).

**Event Escalation**
The staged structure a Scenario can move through (e.g., Planning → Travel → Preparation → Execution → Reaction → Escape → Investigation), each stage having its own probabilities and possible dispatch triggers.

**AI Director / Event Director**
The core decision-making system that scores candidate activities against current World State to decide what an Organization plausibly does next. Produces a legible "why" for each decision (used by the Live World Inspector).

**Heat**
A measure of how much attention/pressure an Organization is currently under. In v1.0, a single scalar per organization; longer-term, may be split into categories (Police / Media / Federal / Gang / Civilian Fear — see `Goals.md`, deferred).

**Intelligence (Police Intelligence)**
A 0–100 score per Organization representing how much law enforcement currently knows about it (aliases, vehicles, safehouses, associates). Rises through arrests/raids/surveillance/informants; decays if the organization successfully hides, destroys evidence, or intimidates witnesses.

**Evidence**
A physical or digital trace left behind by a Scenario (shell casing, blood, fingerprint, phone, tire track, etc.). Has a lifecycle — it can degrade or disappear over time, weather, or foot traffic.

**Investigation Graph**
The set of links connecting pieces of Evidence to suspects, vehicles, associates, and ultimately an Organization. Built incrementally as evidence is recovered and connected.

**Dispatch**
The generic internal representation of a call generated by the simulation, independent of any specific third-party CAD/MDT resource. Converted into a resource-specific format by an Adapter.

**Adapter**
A translation layer that converts DCE's generic Dispatch/Evidence/MDT events into the format expected by a specific third-party resource (e.g., ERS Dispatch, Sonoran CAD, ps-mdt). DCE core never talks to a third-party resource directly — always through an adapter.

**Service**
An independently-registered subsystem (Dispatch, Evidence, Economy, Territories, etc.) that other code accesses only through the Service Registry, never by referencing its internals directly.

**Service Registry**
The mechanism by which Services register themselves and are looked up by other code (`DCE:GetService("Dispatch")`), avoiding hardcoded dependencies between modules.

**Event Bus**
The internal publish/subscribe system all Services use to communicate (e.g., "Organization Started Drug Deal" → picked up by Dispatch, Evidence, Civilian AI, Analytics, etc. independently).

**Plugin**
A self-contained, independently loaded addition to DCE (new Organization type, new Adapter, new Scenario pack, new Behavior) that uses only the public SDK — never edits DCE core files. Declares its own manifest (name, version, dependencies, what it provides).

**SDK**
The documented set of exports, interfaces, and examples that let third-party developers build Plugins and Adapters without modifying DCE core.

**Specification (spec)**
An immutable-until-revised engineering document in `/specifications/` that defines exactly how a system works (its "constitution"), as opposed to `/docs/`, which explains and contextualizes.

**ADR (Architecture Decision Record)**
A recorded justification for a significant or exception-making architectural choice, stored in `/architecture/` — problem, alternatives considered, decision, consequences.

**Simulation Layers (0–3)**
The four fidelity tiers used to keep performance manageable:
- **Layer 0 — Statistical:** whole map, no NPCs, numbers only.
- **Layer 1 — Ambient:** near players, flavor NPCs/behavior, not yet interactive incidents.
- **Layer 2 — Interactive:** player-affectable events (drug deals, thefts, convoys).
- **Layer 3 — Major Incident:** full AI/dispatch/evidence fidelity for escalated events.

**World Chronicle**
(Deferred past v1.0.) A persistent, searchable log of significant world events, intended to give the server an accumulating visible history.

**Materialize**
The act of spawning full NPCs/vehicles/detail for a Layer 0/1 simulated entity when a player comes within range, based on its current simulated state.
